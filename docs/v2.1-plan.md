> Last updated: 2026-05-24 · Owner: Luis · Target ship: 2026-06-01 (1 working
> day of implementation + a release day).
>

## Goal

Ship `nationid/jp` with two specs (`JP_MY_NUMBER`, `JP_CORPORATE_NUMBER`)
following the exact shape of `nationid/in`. Both at `confidence: "high"`.
Adds ~1 country, +2 codes, ~+60 tests; tarball delta ≈ +10 KB.

Why Japan first (vs SG/KR/TW): smallest scope (2 specs), both with
first-party citations and a maintained python-stdnum oracle after the
research-doc fixes ([`v1.2-asia-research/VERIFICATION.md` §JP-1 / §JP-2](./v1.2-asia-research/VERIFICATION.md#japan)).
Lowest correctness risk.

## Out of scope (defer)

- `JP_PASSPORT` (no public check digit beyond ICAO 9303 MRZ).
- `JP_DRIVERS_LICENSE` (12-digit format, no published algorithm).
- Korea / Singapore / Taiwan — separate releases (v2.2 SG, v2.3 KR, v2.4 TW).

## Plan (executable steps)

### Step 1 — Pre-flight (research patches)

Apply the [VERIFICATION.md §JP-1 / §JP-2 deltas](./v1.2-asia-research/VERIFICATION.md#japan)
mentally; the JSDoc on each spec will encode them:

- `JP_MY_NUMBER` JSDoc statute line: `平成26年総務省令第85号 第5条` (full title is the long form recorded in VERIFICATION.md).
- `JP_CORPORATE_NUMBER` JSDoc statute line: `法人番号の指定等に関する省令 第3条`.
- python-stdnum module references: `stdnum/jp/in_.py` and `stdnum/jp/cn.py` (not `corporate_number.py`).

### Step 2 — Implement specs

Mirror `src/countries/in/` structure:

```
src/countries/jp/
  index.ts           re-exports + JP_MY_NUMBER, JP_CORPORATE_NUMBER aliases
  my_number.ts       12-digit, weighted mod-11, r ≤ 1 → 0
  corporate_number.ts 13-digit, weighted mod-9 (NTA scheme), check digit at pos 1
```

Algorithm constants:

```ts
// my_number.ts
const WEIGHTS_LR: readonly number[] = [6, 5, 4, 3, 2, 7, 6, 5, 4, 3, 2];
// check = 11 - (sum % 11); if check ≥ 10 → 0

// corporate_number.ts
// Right-to-left weights (1, 2, 1, 2, 1, 2, 1, 2, 1, 2, 1, 2) over positions 2..13
// check at position 1 = 9 - (S mod 9)
```

Tests in `tests/countries/jp.test.ts` with:

- ≥6 valid synthetic vectors per spec (including the python-stdnum
  doctest samples `621498320257` and `5835678256246`).
- Boundary cases for `JP_MY_NUMBER`: `40000000005 → 0`, `00000000000 → 0`.
- Invalid-checksum vectors, all-same-digit rejection (if any), too short /
  too long.
- Oracle agreement test that re-derives the check digit for 10 000 random
  bodies and matches the python-stdnum algorithm port (pure TypeScript
  port; the test is offline).

### Step 3 — Wire `nationid/jp` subpath

- `package.json` exports: add `"./jp"` mirroring `./in` shape with

- `tsup.config.ts`: add `src/countries/jp/index.ts` as an entry.

## Acceptance criteria

- [ ] `pnpm verify` clean (lint + typecheck + test + build + test:dist + publint).
- [ ] Governance test passes for `JP_MY_NUMBER` and `JP_CORPORATE_NUMBER` (cites the corrected statute strings).
- [ ] python-stdnum oracle agreement test runs 10 000 random samples without failure.
- [ ] `docs/countries/jp.md` published, mirroring `in.md` structure.
- [ ] README counts incremented in all three languages.
- [ ] No regression in existing 52 countries (post-v2.0 EU-VAT baseline of ~6900 tests still pass).

## Risks + mitigations

- **NTA algorithm publication is in ordinance text, not a clean PDF.**
  Mitigation: cross-validate with python-stdnum `cn.py` doctests + the
  live NTA registry resolution of `7000012050002` → "国税庁".
- **My Number is sensitive PII under the My Number Act.** Mitigation:
  ensure `mask` / `hash` / `lastN` work end-to-end before release; add JP
  examples to `docs/PII_GUIDANCE.md`.
