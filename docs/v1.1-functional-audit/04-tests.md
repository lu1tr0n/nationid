# v1.1 Functional Audit — Test posture

## Score: 9/10

`nationid@1.1.0` ships one of the most disciplined test suites I have audited in a small TypeScript library. The pyramid is inverted in a defensible way (heavy unit + property + cross-validation, near-zero E2E because the library has no I/O), test names read like a spec, fixtures are synthetic-but-checksum-correct, and the "governance" precedent (`confidence-citations.test.ts`) is genuinely novel — tests that police rules about rules. 70 files / 6488 tests pass in 127s locally on a single thread, with zero `.skip` or `.todo` outside the cross-validation oracle gates (which auto-skip cleanly when `python3` is unavailable). The remaining point is held back because Stryker / mutation testing is acknowledged as backlog but not yet wired up — without it the suite size alone cannot prove tests would fail under mutation. Once Stryker (or `cargo-mutants`-equivalent for TS) lands and a divergence ledger is added to `docs/CROSS_VALIDATION.md`, this is a 10.

## Strengths

### S1 — Test names read like a specification
`describe("MX — RFC Persona Física") → describe("validate") → it("rejects forbidden 4-letter prefixes (palabras altisonantes)")` is what a maintainer reads when SAT publishes a new forbidden term. No `test_get_user_4` anti-patterns. The Spanish/Portuguese assertion strings include the actual user-facing message verbatim (`"El DUI es demasiado corto."`) — when an i18n string changes you see the diff immediately, and orthography regressions are pinned by string equality.

### S2 — Fixtures are synthetic AND checksum-correct
Every fixture body is generated by mulberry32 keyed on the document code (`tests/cross-validation/_helpers.ts`). The check digit is computed from the **published spec algorithm** re-implemented in-helpers — explicitly *not* borrowed from the reference library — so cross-validation agreement is a real signal, not circular. This is the single most important property of the test suite: a CPF body produced by `generateValidCpfs(60)` is real-shaped enough to fool both `brazilian-utils` and `python-stdnum`, but no real-person digits ever appear. The `synthetic-only guarantee` comment is restated in every generator.

### S3 — Property tests have explicit invariants (P1-P10)
The seven property files in `tests/property/` map 1:1 to numbered invariants documented in `docs/PROPERTY_TESTS.md`. Every property iterates `listSupportedCodes()` so adding a country adds ~7 property tests for free. The fixed seed (`PROPERTY_TEST_SEED = 0x6e6164_6964`) means a shrunk counter-example in local dev reproduces in CI deterministically — a critical detail most fast-check users miss. `parse-success-invariants.test.ts` enforces six sub-invariants per success (`r.normalized === spec.normalize(x)`, `spec.validate(r.formatted)`, etc.) — these are the kind of contract-level assertions that would normally be in a separate type-level test pass.

### S4 — Governance test as policy-as-code
`tests/governance/confidence-citations.test.ts` is the standout file in the repo. It auto-discovers all `src/countries/<cc>/*.ts`, filters to those declaring `confidence: "high"`, and asserts the file-header JSDoc contains either a first-party issuer URL (suffix-walk over a 41-domain allowlist + 11 government-TLD regex patterns) or a recognized legal statute reference (23 statute patterns including `RG AFIP`, `Real Decreto`, `SAT Anexo`, `IRS Pub.`). It runs via `it.each` so a failing PR sees a diagnostic listing the URLs it *did* find and the requirement it failed. This is the precedent for "tests that police rules about rules" and the right model for the rest of v1.x.

### S5 — Algorithmic primitives have isolated unit tests
`tests/algorithms/mod11.test.ts` exercises `mod11WeightedSum` with hand-computed BR CPF / AR CUIT shapes plus error paths (length mismatch, non-digit at known position, negative weights, readonly tuple acceptance). It also verifies `cycleWeights` does not mutate its input. Because ~25 country specs lean on these two functions, having direct unit tests means a regression surfaces at the primitive level — not as 25 simultaneous country-test failures.

### S6 — Clock injection regression
`tests/extract/mx-rfc-clock.test.ts` is exactly the test you write *after* an audit caught the "2056 time bomb" — it pins both branches of the century disambiguator and explicitly documents the 100-year wraparound at the boundary year. The default-clock smoke check (`expect([1956, 2056]).toContain(dob?.year)`) is the right concession: it stays wired without becoming a wall-clock dependency.

### S7 — Dist test runs against the published exports map
`tests/dist/exports.test.ts` is small (121 LOC) but it catches the entire class of "ESM/CJS interop broken in the dist" bugs by `await import("nationid")` and `await import("nationid/<cc>")`. The companion vitest config (`vitest.dist.config.ts`) is excluded from the default test runner so the suite stays fast in development and only runs in `pnpm verify` / CI after `pnpm build`. The dist test verifies (a) root API surface (7 functions present), (b) a known SV_DUI validates through the published path, (c) every country subpath loads + exposes a `*Bundle` export.

### S8 — Cross-validation strategy is multi-source per country where it matters
Brazil docs are validated against TWO independent JS libraries (`@brazilian-utils/brazilian-utils` + `cpf-cnpj-validator`) AND `python-stdnum.br.cpf/cnpj` via the oracle. ES_DNI is validated against `validator.js` AND `python-stdnum.es.dni`. Where the libraries diverge (AR CUIT prefixes 25/26, validator.js silently rewriting dv=10 → 9), the divergence is *documented in the test file header itself* (`tests/cross-validation/validator-js.test.ts` lines 17-30) — not buried in a separate doc. The `CUIT_PREFIXES_NATIONID_ONLY` constant carries the "intentional divergence" assertion test.

### S9 — Oracle gracefully degrades
`tests/cross-validation/_stdnum-oracle.ts` shells out to `python3 -c` with caching, input validation (rejects `"`, `\`, `\n`), and a 5s timeout. On a missing python3 it returns `{ ok: false }` and the suite uses `describe.skip` — visible in test output, no false-green. The `batchStdnumIsValid` helper sends N inputs through a single subprocess (~30ms startup amortized over 40 vectors). This is the right balance — the library does not take a Python runtime dependency, but the gold-standard reference is exercised whenever the dev box has it.

### S10 — Edge cases are exhaustive at the surface
The CURP test (`tests/countries/mx.test.ts`) covers: empty, 17 chars (too short), 19 chars (too long), wrong structure, month 13, day 00, unknown entidad federativa (ZZ), placeholder all-same digits, leap-day boundary (`SANC000229HBCNNN06` — 2000-02-29), and the official RENAPO genérica extranjero (`XEXX010101HNEXXXA4`). NSS test exercises every wrong DV across two bodies (18 explicit rejections per body). This is roughly the level seen across all 34 country files.

### S11 — Coverage thresholds enforced in vitest config
`vitest.config.ts` sets lines 90 / functions 90 / branches 85 / statements 90 via the v8 provider, with `index.ts` re-exports excluded. The threshold actually fails the build if violated — not a soft signal.

### S12 — CI matrix is wide on the right axes
`.github/workflows/ci.yml` runs the entire pipeline (lint, typecheck, test, docs:check, build, test:dist, size) on a 3×3 matrix: `{ubuntu, windows, macos} × {Node 20, 22, 24}`. `fail-fast: false` so a Windows-only break still surfaces. Bundle-size check is correctly limited to a single lane (`ubuntu-latest / Node 22`) to avoid noise. Actions are pinned by commit SHA (supply-chain hygiene).

## Gaps & risks

### Gap 1 — Mutation testing is backlogged, not present
`README.md:218` lists "mutation testing" as v1.1+ backlog. The suite size and the property suite raise the bar, but without Stryker we cannot prove mutants would die. Specifically: a check-digit table off-by-one (the exact B2 bug referenced in `docs/CROSS_VALIDATION.md` § B2) could pass under property tests if the cross-validation generator and the spec share the same flawed table. Property tests pin invariants between `validate` and `parse`; they do not pin `validate` against the *external truth* — only cross-validation does, and that only covers ~14 codes out of 124.

**Risk if not addressed:** A future RFC table tweak, mod-11 weight change, or CURP entidad expansion ships green despite being mathematically wrong, until a real user reports it. The B2 bug was caught precisely because python-stdnum disagreed — the codes WITHOUT an external oracle (SV_DUI, SV_NIT, CR_DIMEX, most BO/EC/PY/VE/PA/UY documents per `docs/CROSS_VALIDATION.md` line 107) are the most exposed.

**Recommendation:** Add `@stryker-mutator/core` + `@stryker-mutator/vitest-runner`, target `src/countries/*/` with `--mutate "src/countries/**/*.ts"` only (not the full codebase), and set a "fix surviving mutants" task for the country files lacking external cross-validation. Stryker against the full 6488 tests will be slow — start with one country per CI run on a `mutation` lane scheduled nightly, not per-PR.

### Gap 2 — Coverage report not enforced in CI
`vitest.config.ts` defines thresholds (90/85/90), but `.github/workflows/ci.yml` runs `pnpm test`, not `pnpm test:coverage`. The thresholds are armed but never fired in CI. Locally, the threshold only catches drops if a contributor remembers to run `test:coverage`.

**Risk if not addressed:** Coverage can silently drop on PRs (e.g. a new country lands without a country test file). The threshold provides false confidence.

**Recommendation:** Add a `coverage` step in `ci.yml` on a single matrix lane (`ubuntu / Node 22`) running `pnpm test:coverage` and uploading to Codecov/Coveralls. Keep the thresholds at 90/85/90 and let CI enforce them.

### Gap 3 — Property tests sometimes vacuously pass (P10, P3 invalid-side)
`tests/property/mask-format-alignment.test.ts:42` itself documents that for checksummed specs, `maskArbitrary` almost never lands on a passing checksum, so P10 is vacuously true in those cases. The file calls this out honestly. Similarly, `parse-success-invariants.test.ts` is only exercised on the success branch via `arbitraryValid(code)` — P3 is never tested on the failure paths. The `numRuns: 100` default is fine for invariants like normalize-idempotency but thin for invariants over the *valid* space when only a fraction of generated inputs pass validation.

**Risk if not addressed:** A subtle bug in `format()` on a checksummed spec might slip through P10 because no shrunk counter-example ever exercises the relevant branch.

**Recommendation:** For the four properties that consume `arbitraryValid(code)` (P3, P5, P6, P7, P10), boost `numRuns` to 250-500. Cost is ~5x — but the suite spends 127s in tests today and 63s of that is `import`. Trial cost is `2.49s tests` total — there is headroom. Alternative: maintain a per-spec curated "edge fixtures" array (leap-day CURPs, all-same-digit DV-10 boundaries, IK letters, etc.) and `it.each` over them in addition to `fc.assert`.

### Gap 4 — No coverage on Bun / Deno / browser runtimes
The matrix is Node 20/22/24 × {ubuntu, windows, macos} only. Library ships as ESM + CJS with subpath exports and is marketed as TypeScript-first; consumers in Bun and modern bundlers (Vite, Next.js, Astro) are common. The dist tests run only on Node.

**Risk if not addressed:** A `subtle.crypto`-vs-`node:crypto` divergence in `pii/hash.ts` (which uses SubtleCrypto) only surfaces on a non-Node runtime. The dist test catches CJS/ESM but not the runtime API surface.

**Recommendation:** Add a Bun lane to `ci.yml` (`oven-sh/setup-bun@v2` action) running `bun run test:dist`. Optional: a `browser` lane via `@vitest/browser` + Playwright that imports the package in a real browser context — overkill for v1.1 but trivial to wire when the React companion lands. This is a v1.2 backlog item.

### Gap 5 — Governance test could regress if a spec deletes `confidence: "high"`
`confidence-citations.test.ts:242` guards with `expect(highSpecs.length).toBeGreaterThan(40)` to catch a vacuous-pass. But the threshold is hard-coded to 40 — if v1.1 added 8 confidence:"high" specs (it did), the floor should rise. Otherwise a future PR that demotes 20 specs to "moderate" would still pass the sanity guard.

**Risk if not addressed:** Quiet drift in the high-confidence tier across releases.

**Recommendation:** Pin the count exactly: `expect(highSpecs.length).toBe(58)` (or whatever v1.1 ships). When the count legitimately changes, the test forces the PR author to update the pin — the right kind of "noisy intentional break". Alternatively, snapshot the list of codes claiming high confidence to a checked-in JSON file and assert equality.

### Gap 6 — Single-process test run, no shardable lane
`pnpm test` runs all 70 files in one vitest process. 127s wall clock locally. CI doesn't shard. On the 3×3 matrix that is 9 parallel ~127s runs, fine for now — but each `import` cost (63s) is per-process overhead from re-resolving 70 spec files + fast-check.

**Risk if not addressed:** As countries are added beyond 34 (roadmap says 9 EU + 12 European + Asia), test wall-clock grows linearly. At ~5 countries per release and ~3s import per country, the v2.0 suite is going to be 4-5 minutes per CI lane.

**Recommendation:** Enable vitest's `--shard` flag and split into 2-3 CI lanes per OS. Or: investigate why `import` takes 63s on a 6488-test suite (likely TypeScript compilation cost via `tsx` — switching to a pre-built fixtures path or using `vitest`'s `optimizeDeps` could halve it).

### Gap 7 — Cross-validation lane fails silently on missing dependencies
The python-stdnum lane uses `describe.skip` with a printed reason when `python3` is unavailable. CI has python3 by default, but the skip is **invisible** in PR checks — you have to read the test output to notice cross-validation was skipped. There is no "expected cross-validation lanes ran" assertion at the CI level.

**Risk if not addressed:** A docker image change, runner upgrade, or pip-install-failure could silently drop the cross-validation coverage and ship green for weeks.

**Recommendation:** Add a single test outside `describe.skip` that asserts `probeStdnum([…]).ok === true` on Linux runners. On Windows/macOS keep the skip path (python-stdnum is harder to install). This is a 5-line change.

### Gap 8 — No fuzz / random-byte stress on `validate`
Property tests use `fc.string({ maxLength: 30 })` which mixes ASCII + control + a few unicode. No test feeds NULL bytes, very long inputs (10 MB string), or adversarial unicode (RTL marks, ZWJ sequences) — which is exactly what a hostile actor would send to a public-facing API consuming `validate`.

**Risk if not addressed:** A regex catastrophic backtrack or unicode normalization bug in `normalize()` could DoS a server-side caller. Low-likelihood (none of the specs use regex with nested quantifiers as far as the read showed), but unproven.

**Recommendation:** Add `tests/property/dos-resilience.test.ts` that generates `fc.string({ maxLength: 100_000 })` and asserts `validate(code, x)` returns within 50ms wall clock. fast-check supports `fc.timeout`.

## Property-testing posture

The 10 invariants ship as documented promises:

| # | Invariant | Where tested |
|---|---|---|
| P1 | `normalize(normalize(x)) === normalize(x)` (idempotency) | `normalize.test.ts` |
| P2 | `parse(x).ok === validate(x)` | `parse-validate-consistency.test.ts` |
| P3 | `parse(x).ok ⇒ 6 sub-invariants hold` | `parse-success-invariants.test.ts` |
| P4 | `ParseError.kind` ∈ {5 known kinds}, exhaustive `never` switch | `parse-validate-consistency.test.ts` |
| P5 | `format(format(x)) === format(x)` for valid x | `format-stability.test.ts` |
| P6 | `normalize(format(x)) === normalize(x)` for valid x | `normalize.test.ts` |
| P7 | Whitespace insertion preserves validity | `whitespace-resilience.test.ts` |
| P8 | `!validate(x) ⇒ !parse(x).ok` | `parse-validate-consistency.test.ts` |
| P9 | `getSpec(spec.code) === spec` per registered code | `registry-self-consistency.test.ts` |
| P10 | `validate(maskSub) ⇒ format(maskSub) === maskSub` | `mask-format-alignment.test.ts` |

**Invariants pinned but vacuous-prone:** P10 (almost always vacuous on checksummed specs — file header acknowledges this), P3-P7 (only exercise the success branch via `arbitraryValid`).

**Invariants NOT yet tested that probably should be:**

- **P11 — `hash(normalize(x)) === hash(x)`** : currently asserted by example in `hash.test.ts:26-41` but not generalized across every code via fast-check. Easy win.
- **P12 — `mask(format(x))` length-stability**: `mask` swears `output.length === mask.length` (`mask.test.ts:125`) for one BR_CNPJ case. Should hold for every code, every valid input, every mask substitution. Easy win.
- **P13 — `validate(x) === validate(format(x))`** for any `x`: today, `format` on invalid input returns the input unchanged (`mx.test.ts:67`), but the property as stated should hold by construction. Worth pinning.
- **P14 — `extract*` consistency with parse**: when `parse(code, x).ok`, the extractor returns the same `region/dob/sex` regardless of input formatting (`12345678-9` vs `12.345.678-9`). Today only exercised by example in country tests.

A `tests/property/pii-invariants.test.ts` consolidating P11/P12/P13 would be the natural next file — modeled on the existing property structure (iterate `listSupportedCodes`, fixed seed, `arbitraryValid` from `_arbitraries.ts`).

## Recommendations

1. **Wire `pnpm test:coverage` into CI** with a Codecov/Coveralls upload — **30 min effort, high leverage**. Adds one step to `ci.yml` on the ubuntu/Node 22 lane. Threshold-driven failure is already configured in `vitest.config.ts`; today it sits unused.

2. **Pin the governance-test high-confidence count exactly** (`expect(highSpecs.length).toBe(58)`) — **5 min effort**. Converts a soft floor into a hard ratchet so demotions trigger a PR diff.

3. **Add Stryker on a nightly scheduled lane against `src/countries/{cc}` files lacking external cross-validation** — **1-2 day effort**. Target: SV, BO, EC, PY, NI, PA, UY, VE, CA, GB, FR, DE, IT, NL, BE, CH, PL, SE, NO, DK, FI, PT — 22 country files. Run with `--mutate "src/countries/{sv,bo,ec,...}/**/*.ts"` to keep runtime bounded. Acceptance criterion: ≥80% mutation score per file, document survivors as `// stryker-ignore` with rationale.

4. **Add P11/P12/P13 property tests** for `hash`, `mask`, and `format` round-trip — **half day effort**. Mirror existing files; reuse `arbitraryValid` from `_arbitraries.ts`.

5. **Add a Bun matrix lane running `test:dist`** — **15 min effort**. Catches non-Node runtime regressions before the React companion ships and Bun-via-Vite consumers start filing issues.

6. **Add `timeout-minutes: 15` to each CI job** — **2 min effort**. Defensive cap; a hung test or vitest 4 import deadlock today burns the default GitHub 6h timeout.

7. **Add an "oracle availability" sanity assertion** outside the `describe.skip` — **5 min effort**. On Linux runners, `expect(probeStdnum(REQUIRED).ok).toBe(true)` so a future regression in the python-stdnum install path fails CI instead of silently skipping.

8. **Consolidate divergences into `docs/CROSS_VALIDATION.md § Divergences`** — **2 hours effort**. Table format: `(spec, source-library, oracle says, nationid says, citation, file:line)`. Becomes the canonical reference for consumers migrating from python-stdnum / validator.js.

9. **Investigate the 63s `import` cost on test startup** — **half day**. Likely tsx compilation overhead. Worth measuring before adding more property invariants since each new file is ~1s of import budget.

10. **Optional: add `numRuns: 250` for properties using `arbitraryValid`** — **15 min effort**. Reduces vacuous-pass risk on P3/P5/P6/P7/P10 for checksummed specs.
