# I18N — error message localization

`nationid` ships a small, hand-written, locale-aware catalog for the
`ParseError` discriminant returned by `parse()`. Everything else
(label resolution for `DocumentSpec.labelKey`, country names, etc.) is
intentionally left to the consumer.

This document covers:

- the locales we ship
- the public API
- how to bundle a single locale (subpath imports)
- the contribution guide for adding a new locale (v0.5 EU plan)
- guidelines for translators

## Supported locales

| Tag  | Language               | Status   | Default? |
| ---- | ---------------------- | -------- | -------- |
| `en` | English                | shipped  | yes      |
| `es` | Spanish (neutral LATAM)| shipped  |          |
| `pt` | Portuguese (neutral)   | shipped  |          |

The default locale is `en`. Unrecognized locale tags fall back to the default.

The list of additional locales targeted for v0.5 (EU bundle release) is
tracked in [the roadmap](../README.md#roadmap). Please open an issue before
starting a new translation so we can avoid duplicate work.

## Public API

```ts
import {
  getErrorMessage,
  getErrorTemplate,
  SUPPORTED_LOCALES,
  DEFAULT_LOCALE,
  type Locale,
} from "nationid/i18n";
```

### `getErrorMessage(error, locale?, documentName?)`

Returns a fully-rendered, human-readable message.

```ts
getErrorMessage({ kind: "too_short" }, "es", "DUI");
// "El DUI es demasiado corto."

getErrorMessage({ kind: "empty" }, "en");
// "Please enter a value."

getErrorMessage({ kind: "invalid_format" }, "pt", "CPF");
// "O formato do CPF não é válido."
```

Arguments:

- `error` — the `ParseError` returned in `ParseResult.reason`.
- `locale` — optional. Defaults to `DEFAULT_LOCALE`. Unsupported tags fall
  back to the default.
- `documentName` — optional. The document name to interpolate (e.g. `"DUI"`,
  `"CPF"`, `"SSN"`). When omitted, the rendered sentence uses neutral phrasing
  ("the document" / "el documento" / "o documento").

Behavior:

- The function never throws.
- Whitespace-only `documentName` is treated as missing and triggers the
  neutral fallback.
- Unknown `kind` values (forward-compat for future `ParseError` variants) get
  a generic per-locale message ("The {document} is not valid.").

### `getErrorTemplate(kind, locale?)`

Returns the raw template **without** interpolation. The string still contains
the literal `{document}` slot for kinds that have one.

```ts
getErrorTemplate("invalid_format", "es");
// "El formato del {document} no es válido."
```

Useful when the consumer has its own templating engine or wants to render
into a structured component (e.g. React with named slots).

### `SUPPORTED_LOCALES`, `DEFAULT_LOCALE`, `Locale`

```ts
const SUPPORTED_LOCALES: readonly Locale[] = ["es", "en", "pt"] as const;
const DEFAULT_LOCALE: Locale = "en";
type Locale = "es" | "en" | "pt";
```

## Bundle one locale only

Each locale is a standalone module, so you can pay only for the strings you
ship. Import the locale directly via the subpath:

```ts
// Spanish only — pulls in ~500 bytes of strings, nothing else.
import { errors as esErrors, neutralDocument } from "nationid/i18n/es";

console.log(esErrors.too_short);
// "El {document} es demasiado corto."
```

Equivalents:

- `nationid/i18n/en`
- `nationid/i18n/es`
- `nationid/i18n/pt`

The orchestrator (`nationid/i18n`) statically imports all three locales.
That is fine for most apps because modern bundlers tree-shake the unused
ones; if your bundler cannot, prefer the per-locale subpaths.

## Adding a new locale

Locale modules are intentionally tiny — five strings plus one neutral noun.

1. Create `src/i18n/locales/<tag>.ts` exporting `errors` and `neutralDocument`
   following the existing files. Use the literal slot `{document}` once per
   template (or omit it for the `empty` template).
2. Add the tag to the `Locale` union and to `SUPPORTED_LOCALES` in
   `src/i18n/index.ts`. Wire the import into `CATALOG` and add an entry in
   `GENERIC_FALLBACK`.
3. Add the subpath export in `package.json` and an entry in `tsup.config.ts`
   (release lead handles this — flag in the PR description).
4. Add a test block in `tests/i18n/errors.test.ts` mirroring the existing
   "documentName interpolation" and "neutral phrasing" suites.
5. Run `pnpm verify` locally before opening the PR.

A native or near-native speaker of the target language must review the
strings. Machine-translated copy is rejected.

## Translator guidelines

- **Tone**: neutral and professional. Avoid colloquialisms and slang.
- **Formality**: address the reader without being condescending. In Spanish
  use the second-person singular informal (`ingresa`, not `ingrese` and
  not `vos ingresá`). In Portuguese, prefer Brazilian neutral phrasing
  (`insira`).
- **Length**: shorter is better — these strings appear inline with form
  fields and tooltips. One sentence per template.
- **Punctuation**: end every template with a period. No exclamation marks.
- **Orthography**: tildes, acentos, cedillas (`ç`), and `ñ` are mandatory.
  Do not strip them "for compatibility" — every consuming runtime supports
  UTF-8.
- **Slot**: use `{document}` exactly once for templates that mention the
  document, and never for `empty`. Do not invent additional slots.
- **No HTML**: templates are plain text. The consumer decides how to render
  emphasis or color.
- **No colons or trailing whitespace**: the consumer concatenates if it
  wants to.

Examples of unacceptable copy:

- `"empty"` → "Por favor, ingresa un valor en el campo del documento de identidad."
  (too long, redundant; ship `"Ingresa un valor."`).
- `"too_short"` → "DOCUMENTO MUY CORTO" (uppercase, no slot, no period).
- `"invalid_checksum"` → "The {document} is invalid because the checksum
  algorithm we use to verify the validity returned a mismatch."
  (way too long; ship `"The {document} is not valid (checksum failed)."`).
